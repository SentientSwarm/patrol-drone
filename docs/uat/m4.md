# M4 UAT — multi-waypoint patrol + abort in SITL

**Goal of this runbook:** one person, one machine, ~5–8 minutes → a machine-checked **PASS/FAIL**
that M4's exit criteria are met (the drone arms, climbs, visits all configured waypoints in order
with a dwell at each, **returns to home**, and lands — and a mid-patrol **external abort** drives an
observable return-home — in PX4 SITL + Gazebo Harmonic).

This is the **M4 slice of the Phase 1 UAT harness**, built on the M3 template ([m3.md](m3.md)). Same
four layers; the M4 additions are the **patrol** route + the **`/patrol/*`** observable surface:

| Layer | Artifact | What it does |
|---|---|---|
| **Doctor** | [`scripts/env_doctor.sh`](../../scripts/env_doctor.sh) | Is this machine *capable*? Fails loud with the exact fix. |
| **Runner** | [`scripts/run_sitl_mission.sh`](../../scripts/run_sitl_mission.sh) | `--patrol` brings the stack up in order, then self-verifies + tears down. |
| **Verifier** | [`scripts/verify_patrol.py`](../../scripts/verify_patrol.py) | Subscribes to the live `/patrol/*` surface, asserts the nominal patrol (AC-2), prints PASS/FAIL. |
| **Runbook** | this file | The step-by-step, incl. the manual external-abort check (AC-6). |

> **Automation lane:** like M3, this is **ROS / launch_testing**-driven (headless flight asserted
> over `/patrol/*` + `/fmu/out/*`). Browser automation (Playwright, `/uat`) is **not** used here —
> that lane is for the web surfaces at M8 (Foxglove).

## Preconditions

- Ubuntu 24.04 host set up via [`scripts/setup_phase1.sh`](../../scripts/setup_phase1.sh) and the ROS
  2 workspace built (`ros2_ws/install/` exists) — same as M3. The Doctor checks all of this.
- Display server is **X11** (unless you run `--no-qgc` fully headless).
- **The patrol route resolves `checkpoint_id`s** (`cp_north`, `cp_east`) against
  [`sim/config/checkpoints.yaml`](../../sim/config/checkpoints.yaml) — an interim 02-authored
  stand-in for a 03 deliverable. The Runner pins the node's `checkpoints_yaml` parameter to that
  repo file, so it resolves regardless of your shell's working directory.

## Step 1 — check the machine is capable

```bash
scripts/env_doctor.sh
```

Apply each `[FAIL]`'s `fix:` until it prints **`all checks passed`**. (The Runner runs the Doctor
for you; this step is optional.)

---

## Step 2 — fly the patrol and self-verify (one command)

```bash
scripts/run_sitl_mission.sh --patrol
```

Brings the stack up in the order that matters, launches the node from `mission_patrol.launch.py`
(with `record:=false` — 05's recorder is not under test here), runs the Verifier, and tears down.
Logs land in `/tmp/patrol-uat/{agent,px4,qgc,node}.log`.

Useful flags (same as M3): `--keep-up`, `--no-qgc`, `--timeout N` (default **300 s** for the
patrol), `--skip-doctor`.

### What you should observe

- In QGroundControl (unless `--no-qgc`): the vehicle **arms**, switches to **Offboard**, **climbs**
  to ~5 m, then flies to each waypoint in turn, **pausing (dwelling)** at each, then **returns to
  home** and **lands/disarms**.
- The Runner prints the acceptance report and exits `0` on PASS:

```
=== M4 patrol-mission acceptance (nominal, AC-2) ===
  [PASS] armed: vehicle reported ARMED
  [PASS] all_waypoints_visited: visited waypoint indices [0, 1, 2, 3] (need 0..3)
  [PASS] dwelled: DWELL observed at a waypoint
  [PASS] returned_home: RTH (return-to-home) observed
  [PASS] landed_disarmed: disarmed after arming + visiting waypoints (landing completed)
=== overall: PASS ===
```

You can watch the live mission surface in another sourced shell:

```bash
ros2 topic echo /patrol/mission_state        # String: ARMING → TAKEOFF → HOVER → WAYPOINT → DWELL → … → RTH → LANDING → DONE
ros2 topic echo /patrol/current_waypoint      # Int32: -1, then 0,1,2,3 at each WAYPOINT/DWELL
```

## Step 4 — external-abort check (AC-6, manual on host)

The nightly SITL test asserts the external abort automatically
([`test_mission_patrol.py`](../../tests/integration/test_mission_patrol.py)). On the host, exercise
it by hand while the patrol is mid-flight (start with `--keep-up`, or in a second sourced shell):

```bash
# while the drone is partway through the patrol (mission_state is WAYPOINT/DWELL/HOVER).
# /patrol/abort is RELIABLE + VOLATILE; `ros2 topic pub -1` waits for the node's subscription
# before publishing, so a plain default publish is QoS-compatible and delivered (no extra flags):
ros2 topic pub -1 /patrol/abort std_msgs/Bool '{data: true}'
```

**Expect:** `/patrol/mission_state` transitions to **`ABORT`** then **`RTH`** — the drone abandons
the remaining waypoints, returns to home, and lands. (Low-battery abort follows the identical
ABORT → RTH path; its SITL trigger is best-effort, so it is unit-tested rather than driven here —
see [`tests/unit/test_abort.py`](../../tests/unit/test_abort.py).)

---

## Notes / gotchas

- **`/patrol/*` are `std_msgs`** (`String` / `Int32` / `Bool`), so MCAP records and Foxglove renders
  them with no custom-type plugin. The two observable outputs (`mission_state` / `current_waypoint`)
  are reliable + transient-local depth-1, so a late subscriber (04 / 05) sees the latest value; the
  inbound `abort` *command* is reliable + **volatile** (a plain `ros2 topic pub` is compatible — the
  abort sticks via the state machine's latch, not topic durability). They are **not** `/fmu/*` — PX4
  never sees them.
- **`DWELL` + the `current_waypoint` index is 04's once-per-checkpoint capture trigger** (OQ-7) —
  the same observable surface, not a separate topic.
- **Every terminal that runs `ros2` against PX4 types needs BOTH sources** (`/opt/ros/jazzy` *and*
  `ros2_ws/install`) — same as M3.

## One source of truth (so host path and CI can't drift)

The patrol PASS/FAIL criteria are defined once in
[`tests/integration/patrol_acceptance.py`](../../tests/integration/patrol_acceptance.py) and
imported by **both** `scripts/verify_patrol.py` (this runbook) and the nightly SITL test
[`tests/integration/test_mission_patrol.py`](../../tests/integration/test_mission_patrol.py) (run by
[`.github/workflows/sitl-nightly.yml`](../../.github/workflows/sitl-nightly.yml), which runs both the
basic and patrol scenarios). There is no second implementation of "what good looks like."
