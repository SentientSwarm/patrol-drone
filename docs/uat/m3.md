# M3 UAT — basic mission (arm → takeoff → hover → land) in SITL

**Goal of this runbook:** one person, one machine, ~5 minutes → a machine-checked **PASS/FAIL** that
M3's exit criterion is met (the drone arms, enters OFFBOARD, climbs to and *settles at* 5 m AGL,
holds, then lands and disarms — in PX4 SITL + Gazebo Harmonic).

This is the **M3 slice of the Phase 1 UAT harness** and the template the later milestones copy. The
harness has four layers:

| Layer | Artifact | What it does |
|---|---|---|
| **Doctor** | [`scripts/env_doctor.sh`](../../scripts/env_doctor.sh) | Is this machine *capable*? Fails loud with the exact fix. |
| **Runner** | [`scripts/run_sitl_mission.sh`](../../scripts/run_sitl_mission.sh) | Brings the stack up in the right order, then self-verifies + tears down. |
| **Verifier** | [`scripts/verify_mission.py`](../../scripts/verify_mission.py) | Subscribes to live telemetry, asserts the criteria, prints PASS/FAIL. |
| **Runbook** | this file | The step-by-step. |

> **Automation lane (set expectations):** M3 verification is **ROS / launch_testing**-driven —
> headless flight behavior asserted over `/fmu/out/*`. Browser automation (Playwright, the `/uat`
> and `/uat-gen` skills) is **not** used here; that lane is for the **web** surfaces that arrive
> later (Foxglove dashboards at M8). Don't reach for Playwright to test SITL flight.

---

## Preconditions

- Ubuntu 24.04 host set up via [`scripts/setup_phase1.sh`](../../scripts/setup_phase1.sh) (ROS 2
  Jazzy, PX4 v1.17.0 at `~/PX4-Autopilot`, Gazebo Harmonic, the uXRCE-DDS agent, QGroundControl).
- The ROS 2 workspace is built: `ros2_ws/install/` exists. If not, the Doctor prints the exact
  build command (strip the uv `.venv` first; do **not** prefix `PYTHONPATH=`).
- Display server is **X11**, not Wayland (Gazebo Harmonic rendering is unreliable under Wayland on
  this hardware).

You don't have to verify these by hand — the Doctor checks all of them.

---

## Step 1 — check the machine is capable

```bash
scripts/env_doctor.sh
```

Every line is either `[ ok ]` or `[FAIL]`; each `[FAIL]` is followed by the exact `fix:` to apply.
The most common one on a fresh host: the agent is installed at `~/.local/bin` but its shared lib
isn't resolvable, so the Doctor prints the `export LD_LIBRARY_PATH=…` line to run. Apply the fixes
until it prints **`all checks passed`**.

(You can skip this step — the Runner runs the Doctor for you and even pre-applies the agent
`PATH`/`LD_LIBRARY_PATH` fix in its own shell.)

---

## Step 2 — fly it and self-verify (one command)

```bash
scripts/run_sitl_mission.sh
```

This brings the stack up **in the order that matters** — the agent must be listening *before* PX4
starts, or the node comes up but gets zero telemetry and the drone never moves:

```
env_doctor → MicroXRCEAgent (udp4 :8888) → PX4 SITL + Gazebo (gz_x500) → QGC → mission node → verify
```

then runs the Verifier and tears everything down. Logs for each component are written to
`/tmp/patrol-uat/{agent,px4,qgc,node}.log`.

Useful flags:

- `--keep-up` — leave the stack running after verifying (inspect in QGC, or re-run the Verifier).
- `--no-qgc` — headless; don't launch QGroundControl (see the QGC note below).
- `--timeout N` — seconds the Verifier watches before giving up (default 120).
- `--skip-doctor` — skip the capability gate.

### What you should observe

- A Gazebo window with the x500 quad (unless `--no-qgc` / headless).
- In QGroundControl: the vehicle **arms**, switches to **Offboard**, **climbs** to ~5 m (a brief
  overshoot toward ~7 m before settling back to 5 m is *normal* step response, not a failure),
  **hovers** ~10 s, then **lands and disarms**.
- The Runner prints the acceptance report and exits `0` on PASS:

```
=== M3 basic-mission acceptance ===
  [PASS] armed: vehicle reported ARMED (arming_state == 2)
  [PASS] offboard: vehicle entered OFFBOARD (nav_state == 14)
  [PASS] reached_altitude: climbed to >= 4.5 m AGL
  [PASS] settled_hover: held 5.0 +/- 0.5 m for 9.x s (need >= 8.0 s)
  [PASS] landed_disarmed: disarmed after arming + climb (landing completed)
=== overall: PASS ===
```

## Step 3 (optional) — verify against an already-running stack

If you started the stack with `--keep-up` (or by hand), run the Verifier standalone in a shell with
**both** environments sourced:

```bash
source /opt/ros/jazzy/setup.bash
source ~/patrol-drone/ros2_ws/install/setup.bash
python3 scripts/verify_mission.py            # add --timeout N to wait longer
```

It prints the same PASS/FAIL report and exits `0` (all passed) / `1` (any failed).

## Notes / gotchas

- **Every terminal that runs `ros2` against PX4 types needs BOTH sources** — `/opt/ros/jazzy` *and*
  `ros2_ws/install`. Missing the second gives `The message type 'px4_msgs/msg/...' is invalid`. The
  Runner does both for you; Step 3 reminds you when running the Verifier by hand.
- **Why QGC?** On an interactive host, PX4 preflight wants a GCS heartbeat or it reports *"No
  connection to the GCS"* and refuses the offboard arm — so the Runner launches QGroundControl by
  default to supply it. The **nightly CI** brings up the identical agent + PX4 SITL *inside a
  container* and the integration test arms headless **without** QGC; `--no-qgc` mirrors that on the
  host (use it only if your SITL params don't require the heartbeat).
- **Output topic names carry `_v1`** (`/fmu/out/vehicle_status_v1`,
  `/fmu/out/vehicle_local_position_v1`): PX4 v1.17's uXRCE-DDS bridge suffixes a topic with `_v{N}`
  only when its `MESSAGE_VERSION >= 1`. The offboard *input* trio is unversioned, so those are the
  bare names. The Verifier reads the `_v1` outputs.

## One source of truth (so the host path and CI can't drift)

The PASS/FAIL criteria are defined once in
[`tests/integration/mission_acceptance.py`](../../tests/integration/mission_acceptance.py) and
imported by **both** `scripts/verify_mission.py` (this host runbook) and the nightly SITL test
[`tests/integration/test_mission_basic.py`](../../tests/integration/test_mission_basic.py) (run by
[`.github/workflows/sitl-nightly.yml`](../../.github/workflows/sitl-nightly.yml)). The bring-up
*order* mirrors [`docker/sim/entrypoint.sh`](../../docker/sim/entrypoint.sh), and the node is
launched from the same `mission_basic.launch.py` the nightly uses. There is no second
implementation of "what good looks like."

## For M4–M8

Each later milestone adds **its own UAT slice** as part of its Definition of Done — extend the
Doctor with any new capability it needs, add the milestone's acceptance criteria to a shared
acceptance module, and write a one-page runbook like this one. Pick the lane that matches the
surface: **ROS / launch_testing** for flight/behavior (M4 multi-waypoint patrol, M5 perception),
**Playwright / `/uat`** for the web surfaces (M8 Foxglove replay/dashboards).
