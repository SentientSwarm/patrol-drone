# sim — Simulation assets

Gazebo Harmonic worlds, models, and PX4 SITL overrides for the Phase 1 patrol. M5 lands the custom
patrol world, the AprilTag checkpoint markers, the shared checkpoint config, the camera-equipped
airframe, and the tooling that ties them together.

## Contents

```
sim/
├── config/
│   └── checkpoints.yaml                  # SHARED source of truth: checkpoint positions + tag ids (Tenet 2)
├── worlds/
│   ├── patrol_world.template.sdf         # hand-authored world (terrain + obstacles + marker placeholder)
│   └── patrol_world.sdf                  # GENERATED by compose_world.py (checked in) — do not hand-edit
├── models/
│   └── apriltag_36h11_<id>/              # GENERATED tag markers (model.config + model.sdf + texture)
├── px4_sitl_overrides/
│   └── gz_x500_patrol/                   # camera-equipped x500 variant (RGB camera -> ROS image topics)
└── tools/
    ├── compose_world.py                  # World Composer: checkpoints.yaml -> patrol_world.sdf (+ drift check)
    └── gen_apriltag_models.py            # AprilTag model/texture generator (pure stdlib)
```

## Conventions

- Worlds are SDF format (`.sdf`). Models follow Gazebo's model directory structure
  (`model.sdf` + `model.config` + `materials/`).
- **No large binary meshes** — simple primitives only (boxes, cylinders, spheres) or small raster
  textures. Keeps physics near-real-time on the Phase 1 minimum spec (H1).
- **No host-specific or absolute paths** (SIM-6). All `<uri>` references are `model://` or
  repo-relative; the composer rejects any non-portable path at generation time.
- The world frame is **world/ENU** (x=East, y=North, z=Up), declared in the world's
  `<spherical_coordinates>`. 02-mission-control owns the ENU→PX4-NED conversion at its boundary.

## Checkpoint config (`config/checkpoints.yaml`)

The single source of truth for where each AprilTag checkpoint sits and how it maps to an id. Owned by
03-sim-environment; consumed by 02 (positions → patrol waypoints) and 04 (tag_id → checkpoint_id).

```yaml
checkpoints:
  - checkpoint_id: "cp_north"            # string, semantic name
    position: { x: 12.0, y: 8.0, z: 1.5 }  # world/ENU, meters
    tag_family: "tag36h11"
    tag_id: 0                            # int; matches sim/models/apriltag_36h11_0
```

Rules: top-level `checkpoints:` key (a list); ≥3 checkpoints, each pair ≥8 m apart, all within the
~40×40 m world (|x|,|y| ≤ 20); `tag_id` is contiguous `0…N-1` and names a model directory. No waypoint
fields (dwell/tolerance) live here — those belong to 02's mission YAML.

### Adding or moving a checkpoint

1. Edit a row in `config/checkpoints.yaml` (or add one with the next `tag_id`).
2. If you added a new `tag_id`, add a canonical 10×10 grid for it to
   `sim/tools/gen_apriltag_models.py` (`CANONICAL_TAG36H11`) and run it (below).
3. Regenerate the world and (if needed) the models:
   ```bash
   python3 sim/tools/gen_apriltag_models.py   # writes sim/models/apriltag_36h11_<id>/
   python3 sim/tools/compose_world.py         # writes sim/worlds/patrol_world.sdf
   ```
4. Commit both the config and the regenerated assets. CI's **world-drift** gate
   (`scripts/check_world_drift.py`) fails the PR if the committed world or models drift from their
   sources — no hand-edit of `patrol_world.sdf` or the model dirs.

## AprilTag markers (`models/apriltag_36h11_<id>/`)

- Family **tag36h11** (the robust `apriltag_ros` default), IDs contiguous `0…N-1`, ~0.5 m edge length.
  This family/ID/size triple is the **hardware-parity contract (SIM-7)**: Phase 4 prints the same tags
  unmodified, and 04-perception's detector targets it directly.
- Textures are the canonical tag36h11 bit grids (from AprilRobotics `apriltag-imgs`, commit
  `f3fd9a7`), upscaled to a crisp grayscale PNG by the pure-stdlib generator. Re-running
  `gen_apriltag_models.py` reproduces the committed bytes deterministically.

## Camera airframe (`px4_sitl_overrides/gz_x500_patrol/`)

A camera-equipped variant of PX4's stock `gz_x500` (an override, not a fork — A2). It merges the
upstream `x500` model and adds a forward, slightly-down RGB camera:

| Property | Value (provisional — OQ-4/OQ-5) |
|----------|----------------------------------|
| Resolution / rate | 640×480 @ 15 Hz |
| Horizontal FOV | ~1.2 rad (~69°) |
| Mount | forward (+x), pitched ~0.35 rad (~20°) down from `base_link` |
| `frame_id` | `camera_link` |

**Image topics (the 04/05 contract):** the gz sensor publishes `gz.msgs.Image` on the gz topic
`/drone/camera/image_raw`; `ros_gz_image image_bridge` (wired in `camera_bridge.launch.py`)
republishes it as ROS 2:

| Topic | Type | Consumer |
|-------|------|----------|
| `/drone/camera/image_raw` | `sensor_msgs/Image` | 04-perception |
| `/drone/camera/image_raw/compressed` | `sensor_msgs/CompressedImage` | 05-logging |

The `/compressed` companion is produced automatically by `image_transport`
(`image_transport_plugins`).

## Running the patrol world (SITL)

The whole stage comes up with one command (the M5 manual check for AC-1 world-load / AC-4 camera /
AC-3 patrol traversal). First make sure the host is ready, then run it:

```bash
# Host capability gate, incl. the M5 camera/world checks (sim assets + drift, PX4 x500 base model,
# M5 launch files installed, ros_gz_image / image_transport_plugins):
scripts/env_doctor.sh --patrol-world

# Bring up patrol_world + gz_x500_patrol + the image bridge, verify the camera publishes, then fly
# the M4 patrol over the checkpoints and verify it visits each. Logs in /tmp/patrol-world-uat.
scripts/run_patrol_world_sitl.sh
scripts/run_patrol_world_sitl.sh --no-patrol   # camera-only smoke (AC-4): skip the patrol
scripts/run_patrol_world_sitl.sh --no-gui      # headless (camera still renders offscreen)
```

**How the bring-up works (and why it differs from `run_sitl_mission.sh`).** PX4 SITL + Gazebo run
outside ROS; only the camera bridge and patrol node are ROS launches. The stock M3/M4 runner uses
`make px4_sitl gz_x500`, which lets PX4 start Gazebo and spawn the stock x500 into the *default*
world. M5 needs a *custom* world **and** a *custom* camera model that isn't a PX4 airframe — and PX4
only spawns models from its own tree via an airframe file (adding one is a PX4 fork, which A2
forbids). So `run_patrol_world_sitl.sh` uses PX4's no-fork **"bring your own model" attach** mode:

1. Export `GZ_SIM_RESOURCE_PATH` (repo `sim/` + PX4's stock models, so the world's `model://` tags and
   the x500 our model merges both resolve) and `GZ_SIM_SERVER_CONFIG_PATH` (PX4's `server.config`, so
   the Gazebo Sensors system that renders the camera loads).
2. Start the Micro XRCE-DDS agent, then **Gazebo standalone** with `patrol_world.sdf`.
3. **Spawn `gz_x500_patrol`** into the running world (gz `EntityFactory` create) at the takeoff origin.
4. Start PX4 with `PX4_GZ_STANDALONE=1 PX4_GZ_MODEL_NAME=gz_x500_patrol PX4_GZ_WORLD=patrol_world` so
   it **attaches** to our model (`PX4_SIM_MODEL=gz_x500` supplies the x500 airframe params).
5. For the patrol path, launch **QGroundControl** to supply the GCS heartbeat PX4 preflight needs to
   arm offboard (otherwise the mission node loops on `Preflight Fail: No connection to the GCS`); the
   camera-only smoke and `--no-gui` headless runs skip it.
6. Launch `camera_bridge.launch.py`, verify `/drone/camera/image_raw` (polling up to `CAMERA_WAIT`
   seconds, since the camera renders in lockstep with PX4 and first-frame lags cold start), then fly
   `mission_patrol.launch.py` against `sim/config/checkpoints.yaml` and verify the traversal.
